# Руководство для создателя: подпись соглашения (Agreement Document)

Этот документ описывает, как создатель (Creator) VAPI OSP может оформить и подписать документ коммерческого соглашения (Agreement Document), чтобы лицензиат мог проверить подлинность разрешения. Соглашение должно быть **документом**, отправленным с **официальной почты** создателя и при необходимости подписанным **криптографической подписью** (например, PGP/GPG).

---

## 1. Зачем нужна подпись

- Лицензиат может убедиться, что разрешение действительно выдал создатель, а не третье лицо.
- Подпись гарантирует, что текст документа не был изменён после отправки.
- В лицензии (раздел 3A) указано: при опубликованном создателем способе проверки подписи (например, PGP) документ должен быть подписан.

Рекомендуется: создать PGP-ключ, опубликовать открытый ключ в CONTACT.txt или в репозитории, подписывать каждое соглашение этим ключом и отправлять документ с официального email.

---

## 2. Что должно быть в документе соглашения

Согласно разделу 3A лицензии, в документе должны быть:

- **Идентификация лицензиата** — имя или юридическое лицо.
- **Объём разрешённого использования** — что именно разрешено (продукт, сфера, территория и т.д.).
- **Дата вступления в силу**.
- **Явная формулировка** о том, что создатель разрешает указанное коммерческое использование Software (VAPI OSP).
- При необходимости — ссылка на отдельный договор или номер соглашения (если вы ведёте учёт).

Документ можно оформить как письмо в теле email или как прикреплённый файл (например, PDF или текстовый файл). Если используется подпись — подписывается итоговый текст/файл.

---

## 3. Как создать свою подпись (PGP/GPG)

### 3.1. Установка GnuPG

- **Windows:** скачайте Gpg4win с https://www.gpg4win.org/ или установите через `winget install GnuPG.GnuPG`.
- **macOS:** `brew install gnupg`
- **Linux:** обычно уже установлен; при необходимости: `sudo apt install gnupg` (Debian/Ubuntu) или аналог.

Проверка: в терминале выполните `gpg --version`.

### 3.2. Создание пары ключей (ключ создателя)

1. Откройте терминал (командную строку) и выполните:

   ```bash
   gpg --full-generate-key
   ```

2. Выберите тип ключа: **RSA and RSA** (по умолчанию), Enter.

3. Размер ключа: **4096** бит (рекомендуется для долгосрочного использования).

4. Срок действия: укажите период (например, 2 года) или **0** (без срока), если так удобнее.

5. Укажите **имя** и **email**. Email должен совпадать с официальным адресом, с которого вы отправляете соглашения (тот, что указан в CONTACT.txt). Дополнительные email можно добавить позже командой `gpg --edit-key <KEY_ID>` → `adduid`.

6. Задайте **пароль (passphrase)** для закрытого ключа. Он нужен при каждой подписи; храните его в безопасном месте.

После этого GnuPG создаст пару: **закрытый ключ** (хранится только у вас) и **открытый ключ** (его можно публиковать).

### 3.3. Узнать ID и отпечаток ключа

Выполните:

```bash
gpg --list-secret-keys --keyid-format=long
```

Пример вывода:

```
sec   rsa4096/ABCD1234EF567890 2026-01-15 [SC]
      XXXXXXXXXXXXXXXX...
uid                 [ultimate] Your Name <your@official-email.com>
```

- **KEY_ID** — часть после `rsa4096/`, например `ABCD1234EF567890`.
- Отпечаток (fingerprint): `gpg --fingerprint YOUR_KEY_ID` — длинная строка, её можно указать в CONTACT.txt для надёжной идентификации.

### 3.4. Экспорт и публикация открытого ключа

Экспорт в формате ASCII (для вставки в файл или на сайт):

```bash
gpg --armor --export YOUR_KEY_ID
```

Скопируйте блок от `-----BEGIN PGP PUBLIC KEY BLOCK-----` до `-----END PGP PUBLIC KEY BLOCK-----` и:

- добавьте его в CONTACT.txt в раздел «PGP public key», или
- сохраните в репозитории, например `licenses/CREATOR_PGP_PUBLIC_KEY.asc`, и укажите путь в CONTACT.txt.

Опционально: загрузите ключ на ключевой сервер, чтобы лицензиаты могли получить его по email или KEY_ID:

```bash
gpg --send-keys YOUR_KEY_ID
```

(По умолчанию ключ уходит на keys.openpgp.org; можно указать другой сервер при необходимости.)

### 3.5. Подписание документа соглашения

**Вариант A: подписать текстовый файл (например, agreement.txt)**

1. Составьте текст соглашения (с данными лицензиата, объёмом использования, датой и явным разрешением).
2. Сохраните в файл, например `agreement.txt`.
3. Создайте подпись:

   ```bash
   gpg --sign --armor -o agreement.txt.asc agreement.txt
   ```

   Будет запрошен пароль от закрытого ключа. В результате появится файл `agreement.txt.asc` — это подписанный документ в текстовом виде. Отправьте лицензиату оба файла: `agreement.txt` и `agreement.txt.asc`, или один объединённый (см. ниже).

**Вариант B: подпись в одном файле (clearsigned)**

```bash
gpg --clearsign agreement.txt
```

Создаётся файл `agreement.txt.asc`, внутри которого и исходный текст, и подпись. Удобно отправлять одним вложением.

**Вариант C: подписать PDF или другой бинарный файл**

```bash
gpg --sign --armor -o document.pdf.asc document.pdf
```

Отправьте лицензиату `document.pdf` и `document.pdf.asc`. Проверка: `gpg --verify document.pdf.asc document.pdf`.

### 3.6. Отправка с официальной почты

- Отправьте подписанный документ (и при необходимости исходный файл) **с того самого email**, который указан в CONTACT.txt как официальный канал создателя.
- В письме можно кратко указать: «Во вложении — подписанное соглашение; проверка подписи: gpg --verify файл.asc [файл]».

Лицензиат сможет проверить подпись вашим открытым ключом и убедиться, что документ отправлен с вашей стороны и не изменён.

---

## 4. Как лицензиат проверяет подпись

1. Скачивает ваш открытый ключ из CONTACT.txt или из репозитория.
2. Импортирует ключ: `gpg --import CREATOR_PGP_PUBLIC_KEY.asc`
3. Для проверки подписанного файла:
   - если подпись в отдельном файле: `gpg --verify agreement.txt.asc agreement.txt`
   - если используется clearsign: `gpg --verify agreement.txt.asc`
4. Если GnuPG сообщает «Good signature» и указывает ваш ключ — подпись действительна.

---

## 5. Краткий чек-лист для создателя

1. Установить GnuPG и сгенерировать ключ с официальным email.
2. Опубликовать открытый ключ в CONTACT.txt (или в `licenses/CREATOR_PGP_PUBLIC_KEY.asc`) и при необходимости на ключевом сервере.
3. В каждом соглашении указывать: лицензиат, объём использования, дату, явное разрешение.
4. Подписывать документ закрытым ключом: `gpg --sign` или `gpg --clearsign`.
5. Отправлять подписанный документ с официального email, указанного в CONTACT.txt.

Так вы обеспечите соответствие требованиям лицензии (раздел 3A): соглашение — это документ, отправленный с вашей почты и при необходимости проверяемый по вашей подписи.
