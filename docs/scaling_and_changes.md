# Руководство по масштабированию и внесению изменений

Документ описывает, как расширять VAPI новыми модулями, бэкендами и приложениями, где размещать код, как править сборку и тесты, и на что обращать внимание при изменении API.

Связанные документы: [architecture.md](architecture.md) (слои и сценарии), [shortcomings.md](shortcomings.md) (ограничения и точки расширения), [quick_reference.md](quick_reference.md) (основные классы).

---

## 1. Структура проекта и зависимости

### 1.1. Каталоги

| Каталог | Назначение |
|---------|------------|
| `include/` | Заголовки по слоям: `core/`, `platform/`, `gpu/`, `resource/`, `render/`, `font/`, `ui/` |
| `src/` | Реализации: один подкаталог на слой, имена `.cpp` обычно совпадают с модулем |
| `examples/` | Примеры использования (отдельные исполняемые файлы) |
| `tests/` | Юнит- и интеграционные тесты |
| `docs/` | Документация; `detail_docs/` — подробный API |
| `third_party/` | Сторонние зависимости (например, для шрифтов) |

### 1.2. Зависимости между слоями (снизу вверх)

```
core          ← базовый слой (типы, ошибки, файлы, интерфейсы)
platform      ← core + GLFW
gpu           ← core + Vulkan (+ glfw для surface)
font          ← core
resource      ← gpu
render        ← gpu
vapi_font_render ← render + resource + font + platform
vapi_ui       ← core
```

- **core** и **font** не зависят от Vulkan.
- Новый код, который не трогает GPU, по возможности опирается на **core** (и при необходимости **font**, **ui**), чтобы не тянуть платформу и Vulkan.

## 2. Добавление нового кода

### 2.1. Новый модуль (библиотека)

1. **Заголовки:** `include/<layer>/<module>.hpp` (или подкаталог `include/<layer>/<sub>/`).
2. **Реализация:** `src/<layer>/<module>.cpp` (при необходимости несколько файлов).
3. **CMake:** в корневом `CMakeLists.txt`:
   - `add_library(vapi_<name> src/<layer>/<file>.cpp ...)`
   - `target_include_directories(vapi_<name> PUBLIC ${CMAKE_CURRENT_SOURCE_DIR}/include)`
   - `target_link_libraries(vapi_<name> PUBLIC vapi_<dep> ...)` по слоям (см. п. 1.2).

Если модуль — часть существующего слоя (например, новый менеджер в resource), добавьте только новые `.cpp` в соответствующий `add_library(vapi_resource ...)` и заголовки в `include/resource/`.

### 2.2. Новый пример (executable)

1. Исходник: `examples/<name>.cpp`.
2. В `CMakeLists.txt`:
   - `add_executable(<target> examples/<name>.cpp)`
   - `target_include_directories(<target> PRIVATE ${CMAKE_CURRENT_SOURCE_DIR}/include)`
   - `target_link_libraries(<target> PRIVATE vapi_<lib1> vapi_<lib2> ...)`
   - Для примеров с шейдерами: `target_compile_definitions(... SHADER_DIR="${VAPI_FONT_SHADER_DIR}")` и при необходимости `add_dependencies(<target> font_shaders)`.
   - При использовании импортируемой GLFW DLL — добавить `add_custom_command(TARGET <target> POST_BUILD ...)` для копирования DLL (по аналогии с `calculator_example`).

### 2.3. Новый тест

1. Исходник: `tests/test_<name>.cpp`.
2. В `CMakeLists.txt` в блоке `set(TEST_SOURCES ...)` добавить `tests/test_<name>.cpp`.
3. В цикле `foreach(t ${TEST_SOURCES})` добавить ветку `elseif(name STREQUAL test_<name>)` с нужными `target_link_libraries(...)` (только те библиотеки, которые реально использует тест).

Запуск: `ctest` или отдельно исполняемый файл `test_<name>`.

## 3. Расширение по слоям

### 3.1. Новый бэкенд платформы (например SDL, Win32, headless)

- **Интерфейс:** код приложения должен зависовать от `IPlatform` (`include/core/interfaces/` через `platform/platform.hpp`), а не от GLFW.
- **Реализация:**
  - Новый класс, реализующий `IPlatform` (окно, ввод, мониторы, Vulkan-расширения).
  - Новый файл в `src/platform/`, например `sdl_platform.cpp`.
- **Фабрика:** в `createPlatform()` или отдельной фабрике по типу платформы возвращать нужную реализацию (сейчас — только GLFW). Добавить опцию (переменная окружения, аргумент, конфиг), если бэкендов станет несколько.
- **Сборка:** новая цель `vapi_platform_sdl` или включение в `vapi_platform` через условную компиляцию/подкаталоги CMake; линковка с SDL вместо/вместе с GLFW не должна ломать существующие примеры.

Рекомендации из [shortcomings.md](shortcomings.md): добавление headless или SDL-бэкенда улучшит CI и проверку абстракции платформы.

### 3.2. Новый GPU-бэкенд (D3D12, Metal)

Текущая реализация — Vulkan-only; задел под расширение уже есть:

- **ABI (backend-agnostic):** `core/interfaces/i_gpu_abi.hpp` — типы форматов, шейдерных стадий, вершинных описаний, топологии, compare op. Используйте их в общем коде, а в бэкенде конвертируйте в нативные типы.
- **Vulkan-конвертация:** `gpu/vk_abi_convert.hpp` — пример маппинга ABI → Vulkan. Для D3D12/Metal понадобятся свои конвертеры.
- **Интерфейс бэкенда:** `IGpuBackend` (device/instance/surface/swapchain как `void*`), `GpuBackendType`. После `type()` приводить к конкретному API.
- **Render/Resource:** сейчас завязаны на Vulkan (VkFormat, VkBuffer и т.д.). Введение второго бэкенда потребует либо ветвления по типу бэкенда, либо слоя абстракции поверх ABI; это выходит за рамки текущего «Vulkan-first» дизайна (см. [shortcomings.md](shortcomings.md)).

Практичный путь: сначала реализовать второй бэкенд в отдельном подмодуле/каталоге, использовать `i_gpu_abi` и свой аналог `vk_abi_convert`, затем выносить общие части в core/render по мере необходимости.

### 3.3. Новое приложение поверх рендера (аналог FontRenderApp)

- **Вариант 1 — отдельный класс «приложения»:** по аналогии с `FontRenderApp`: свой класс в `include/render/` (или отдельный каталог приложений), своя реализация в `src/render/` или `src/app/`. Внутри: платформа, окно, `GpuContext`, `RenderContext`, ресурсы, пайплайны, цикл (beginFrame → draw → endFrame). Инициализация и `run()` инкапсулированы в API класса.
- **Вариант 2 — свой цикл в main:** как в примере с `RenderContext` в [detail_docs/render.md](detail_docs/render.md): создаёте платформу, окно, `GpuContext`, `RenderContext`, BufferManager/ImageManager и т.д., в цикле вызываете `beginFrame`, `beginRendering`, draw, `endRendering`, `endFrame`; при resize — `recreateSwapchain` и `handleResize`.

Для «масштабирования» количества сцен или рендер-целей достаточно добавлять новые пайплайны и проходы внутри существующего `RenderContext` или своего контейнера, не меняя базовый API контекста.

### 3.4. Расширение модуля font

- **Новый источник шрифтов:** реализовать интерфейс из `font/i_font_source.hpp`, зарегистрировать/подключить в сервисе шрифтов. Размещение: `include/font/`, `src/font/`.
- **FontDrawList:** опции и форматы вывода заданы в заголовке и типах; при добавлении новых опций (например, другой формат атласа) — расширить параметры в `font_draw_list.hpp` и реализацию в `font_draw_list.cpp`, не ломая существующие вызовы (значения по умолчанию).
- **TextPainter:** то же — новые опции через параметры с дефолтами; при смене контракта layout/paint — обновить примеры и тесты в `tests/test_text_painter_*.cpp`, `tests/test_font_*.cpp`.

### 3.5. Новые ресурсы и пайплайны

- **Ресурсы:** новые типы буферов/изображений — по возможности через существующие `BufferManager`/`ImageManager`; при необходимости нового вида ресурса (например, акселераторы) — новый менеджер в `resource/` с зависимостью от `vapi_gpu`.
- **Пайплайны:** новые шейдеры и `GraphicsPipelineBuilder`/`ComputePipelineBuilder` — в `render/`; конфигурация через существующие дескрипторы и layout’ы. Кеш пайплайнов: см. [pipeline_cache.md](pipeline_cache.md).

## 4. Изменения API и контрактов

### 4.1. Интерфейсы (core/interfaces/)

- Изменение сигнатур или семантики интерфейсов затрагивает все реализации и вызывающий код. Рекомендация из [shortcomings.md](shortcomings.md): при появлении реальных бэкендов при необходимости **уточнять** интерфейсы, а не только реализовывать «как есть».
- Добавление новых методов: по возможности делайте их опциональными (например, с дефолтной реализацией в базе или отдельным интерфейсом-расширением), чтобы старые реализации не ломались.
- Именование: в документации и коде используется **IPlatform**; алиас **IPlatformBackend** сохранён для совместимости.

### 4.2. Публичные заголовки (include/)

- Удаление или переименование публичных символов — ломающий изменение; при необходимости сохраняйте алиасы или устаревшие имена с пометкой `[[deprecated]]` на один-два релиза.
- Новые параметры в функциях: задавайте значения по умолчанию, чтобы старый код продолжал собираться.

### 4.3. Ошибки и Result

- Новые коды ошибок добавляйте в соответствующий enum в `core/error.hpp` и при необходимости в `errorMessage()` (см. [detail_docs/core.md](detail_docs/core.md)). Не переиспользуйте существующие коды с другой семантикой.

## 5. Сборка и тесты

### 5.1. CMake

- Зависимости между целями задавайте явно: только те библиотеки, которые реально используются (см. раздел тестов в `CMakeLists.txt`).
- Для примеров с Vulkan/GLFW не забывайте копирование DLL при использовании импортируемого GLFW (см. п. 2.2).
- Шейдеры: путь к скомпилированным SPIR-V задаётся через `SHADER_DIR`; при добавлении новых шейдеров добавьте соответствующие `add_custom_command` и при необходимости общий target по аналогии с `font_shaders`.

### 5.2. Тесты

- Тесты, требующие дисплея (окно, Vulkan surface), в CI обычно запускают с `--headless` где поддерживается; см. [testing.md](testing.md).
- При добавлении нового модуля добавьте хотя бы один тест (юнит или интеграционный), чтобы регрессии были видны при изменениях.

## 6. Чек-лист при внесении изменений

- [ ] Код размещён в правильном слое и не тянет лишних зависимостей (см. п. 1.2).
- [ ] Новые цели добавлены в `CMakeLists.txt`; линковка и зависимости указаны корректно.
- [ ] Публичный API по возможности обратно совместим (дефолты, алиасы, deprecated вместо удаления).
- [ ] Обновлена документация: [quick_reference.md](quick_reference.md), при необходимости [architecture.md](architecture.md) и [detail_docs/](detail_docs/).
- [ ] Добавлены или обновлены тесты; при изменении поведения — обновлены ожидаемые результаты.
- [ ] Учтены ограничения и рекомендации из [shortcomings.md](shortcomings.md) (платформа, GPU, батчинг, headless CI и т.д.).

| Задача | Где смотреть | Действия |
|--------|---------------|----------|
| Новый модуль/библиотека | п. 2.1, CMakeLists.txt | include + src + add_library + target_link_libraries |
| Новый пример | п. 2.2 | examples/*.cpp + add_executable + линковка + SHADER_DIR/DLL при необходимости |
| Новый тест | п. 2.3 | tests/test_*.cpp + TEST_SOURCES + ветка link в CMake |
| Новый бэкенд платформы | п. 3.1, platform/, IPlatform | Реализация IPlatform, фабрика, условная сборка |

| Новое приложение (как FontRenderApp) | п. 3.3 | Класс в render/ или app/, init + run + ресурсы |
| Расширение font | п. 3.4 | i_font_source, FontDrawList/TextPainter с дефолтами |
| Новые ресурсы/пайплайны | п. 3.5 | resource/, render/, pipeline_cache |
| Изменение API | п. 4 | Интерфейсы, дефолты, deprecated, error codes |
| Сборка/тесты | п. 5–6 | CMake, ctest, headless, чек-лист |
