# Слой безопасности VAPI

VAPI — библиотека в одном процессе с приложением. В отличие от Chromium, нет песочницы, отдельного GPU-процесса или принудительного IPC. Безопасность и изоляция лежат на стороне приложения и ОС; этот документ описывает **стек слоя безопасности**: критические места, методы их защиты и общие практики кибербезопасности.

---

## 1. Критичные места (Attack Surface)

### 1.1 Файловая система и пути

| Место | Файлы | Риск |
|-------|--------|------|
| **Чтение по пути** | `src/core/file_utils.cpp` — `readText`, `readBinary`, `readLines`, `readTextLimited`, `FileByteSource` | Путь передаётся как есть; нет канонизации и проверки выхода за пределы разрешённой директории. При пользовательском/сетевом пути возможна **path traversal** (`../`, симлинки). |
| **Запись по пути** | `file_utils.cpp` — `writeText`, `writeBinary`, `FileByteSink`, `createDirectories`, `copyFile` | Аналогично: запись в произвольный путь может перезаписать системные файлы или конфиги. |
| **Кэш шейдеров** | `src/core/shader_cache_file.cpp` — `FileShaderCache::getOrCompile(key, ...)` | Путь формируется как `basePath / key` или `key`; **key** под контролем вызывающего — возможен путь вне basePath при некорректном basePath или злонамеренном key. |
| **Загрузка шейдеров** | `src/render/shader.cpp` — `ShaderModule::loadFromFile(device, path, stage)` | Путь используется напрямую в `std::ifstream`; нет проверки, что файл внутри разрешённого каталога. |
| **Логгер** | `src/core/logger.cpp` — `fopen(m_filepath, "a")` | Путь к лог-файлу из конфига без санитизации; при скомпрометированном конфиге — запись в произвольное место. |

**Итог:** Любой входной путь (от пользователя, конфига, сети) без проверки — критичная точка.

---

### 1.2 Шейдеры и GPU

| Место | Файлы | Риск |
|-------|--------|------|
| **SPIR-V из файла** | `src/render/shader.cpp` — `loadFromFile` → `loadFromMemory` → `vkCreateShaderModule` | Проверяется только выравнивание размера по 4 байта. **Нет валидации содержимого SPIR-V** (драйвер валидирует, но злонамеренный байткод может эксплуатировать баги драйвера или вызывать DoS). |
| **SPIR-V из кэша** | `src/core/shader_cache_file.cpp` — загрузка по пути, затем `vkCreateShaderModule` | То же: подмена файла в кэше — выполнение произвольного SPIR-V в контексте приложения. |
| **loadFromMemory** | `src/render/shader.cpp` | Принимает `std::span<const u32>` без проверки размера/ограничения; очень большой буфер может привести к DoS или переполнению в драйвере. |

**Итог:** Недоверенный или непроверенный SPIR-V — критичная точка (утечка данных, DoS, потенциально RCE через драйвер).

---

### 1.3 Буферы и память GPU

| Место | Файлы | Риск |
|-------|--------|------|
| **createBuffer** | `src/resource/buffer_manager.cpp` | Размер и флаги из вызывающего кода; нет верхнего лимита размера — возможен **исчерпание памяти GPU/хоста**. |
| **map** | `src/resource/buffer_manager.cpp` — `BufferManager::map(id, offset, size)` | **offset и size не проверяются** на границы буфера (`offset + size <= buf.size`); передаются в `vkMapMemory` — риск выхода за границы или некорректного использования. |
| **write** | `src/resource/buffer_manager.cpp` — `BufferManager::write(id, data, size, offset)` | Проверка `offset + size > buf.size` есть — границы соблюдаются. |
| **uploadBuffer / uploadImage** | `src/resource/staging.cpp` | `uploadBuffer(dst, data, dstOffset)` не проверяет, что `dstOffset + data.size()` не превышает размер `dst`; возможна **перезапись за границей** буфера назначения. |
| **updateImage** | `src/resource/staging.cpp` | Есть проверка `data.size() < width*height*4`; размеры width/height от вызывающего — при ошибочных значениях возможны артефакты или переполнение. |

**Итог:** Невалидированные offset/size при map и копировании — переполнение буфера на GPU или краш.

### 1.4 Платформа и ввод

| Место | Файлы | Риск |
|-------|--------|------|
| **Окно и ввод** | `src/platform/glfw_platform.cpp` | GLFW — доверенная библиотека; риски в основном от логики приложения (инъекция через ввод в UI). |
| **Логгер и ОС** | `src/core/logger.cpp` | Чтение `GetComputerNameA`, `GetAdaptersAddresses` (Windows), `/sys/class/net/...` (Linux) при детальном логировании — утечка информации о хосте в лог. |

**Итог:** Логирование чувствительных данных и путь к лог-файлу — второстепенные, но важные точки.

### 1.5 Инициализация и зависимости

| Место | Риск |
|-------|------|
| **Vulkan instance/device** | Включение слоёв (например, validation) по конфигу; отключение в релизе — меньше проверок. |
| **Внешние библиотеки** | GLFW, Vulkan SDK — уязвимости в зависимостях влияют на всё приложение. |

## 2. Методы защиты критичных мест

### 2.1 Пути и файловая система

| Метод | Описание | Где применить |
|-------|----------|----------------|
| **Канонизация и jail** | Перед любым открытием файла: `file::canonicalize(path)`, затем проверка, что результирующий путь **начинается с** разрешённого корня (например, `allowedRoot` для ассетов/шейдеров). Отклонять симлинки вне корня. | `file_utils.cpp` (обёртки `readSecure`/`writeSecure` или опция), `shader.cpp::loadFromFile`, `shader_cache_file.cpp::getOrCompile`, инициализация логгера. |
| **Allowlist расширений** | Для шейдеров — разрешать только пути с суффиксом `.spv` (или явный список). Для кэша — только имена без `/` и `..`. | `ShaderModule::loadFromFile`, `FileShaderCache::getOrCompile`. |
| **Ограничение ключа кэша** | В `FileShaderCache` ключ не должен содержать путь (`/`, `\`, `..`). Нормализовать key к имени файла (stem + hash) или отклонять недопустимые ключи. | `shader_cache_file.cpp`. |
| **Безопасный путь логгера** | При инициализации логгера проверять путь через canonicalize + jail в директорию логов приложения. | `logger.cpp`. |

**Реализация (эскиз):**

```cpp
// Допустим, есть allowedRoot (fs::path) для ресурсов
Result<fs::path> resolveSecure(const fs::path& path, const fs::path& allowedRoot) {
    auto canon = file::canonicalize(path);
    if (!canon) return canon.error();
    const auto canonStr = canon->string();
    const auto rootStr = allowedRoot.string();
    if (canonStr.size() < rootStr.size() || canonStr.compare(0, rootStr.size(), rootStr) != 0)
        return std::unexpected(errors::file::PermissionDenied);
    return *canon;
}
```

### 2.2 Шейдеры (SPIR-V)

| Метод | Описание | Где применить |
|-------|----------|----------------|
| **Лимит размера** | Ограничить размер SPIR-V (например, 1–4 MiB) перед вызовом `vkCreateShaderModule`. | `ShaderModule::loadFromMemory`, загрузка из кэша. |
| **Валидация SPIR-V** | Использовать официальный валидатор (например, `spirv-val` из SPIRV-Tools) перед созданием модуля; в релизе можно отключать для производительности, но для недоверенных шейдеров — обязательно. | Отдельный шаг перед `loadFromMemory` или внутри `loadFromMemory` при опции `validateSpirv`. |
| **Подпись/хеш кэша** | Хранить в кэше не только SPIR-V, но и хеш (SHA-256) исходника/опций; при загрузке проверять целостность. Защита от подмены файла в кэше. | `FileShaderCache`: формат кэша с полем hash, проверка при чтении. |
| **Недоверенные шейдеры в отдельном контексте** | Если приложение загружает шейдеры от пользователя (моды, плагины), по возможности создавать отдельный VkDevice или ограничивать возможности (например, без доступа к чувствительных ресурсов). | Архитектура приложения. |

---

### 2.3 Буферы и копирование

| Метод | Описание | Где применить |
|-------|----------|----------------|
| **Проверка границ в map** | В `BufferManager::map(id, offset, size)` проверять `offset + size <= buf.size` и `offset < buf.size`. При несоответствии возвращать `errors::InvalidArgument`. | `buffer_manager.cpp`. |
| **Проверка границ при копировании** | В `StagingManager::uploadBuffer(dst, data, dstOffset)` требовать от вызывающего размер буфера `dst` или хранить его в обёртке; проверять `dstOffset + data.size() <= dstSize`. | `staging.cpp`, либо API с `BufferId` вместо сырого `VkBuffer` и размером из менеджера. |
| **Лимиты размера буфера** | В `createBuffer` отклонять запросы с `desc.size` выше заданного максимума (например, 256 MiB на буфер) или суммарно по менеджеру. | `buffer_manager.cpp`, конфиг. |
| **Валидация размеров изображений** | В `updateImage`/`uploadImage` проверять разумные лимиты width/height (например, не более 16K) и согласованность с `data.size()`. | `staging.cpp`. |

---

### 2.4 Логирование и конфиденциальность

| Метод | Описание |
|-------|----------|
| **Не логировать чувствительные данные** | Не писать в лог имена хостов, MAC, IP, пути к пользовательским файлам в режиме, доступном третьим лицам. Либо отключать детальное сетевое/системное логирование в релизе. |
| **Фиксированный каталог логов** | Путь к лог-файлу — только под разрешённой директорией приложения (jail). |

## 3. Методы кибербезопасности (общий стек)

### 3.1 Принципы

- **Минимальные привилегии:** библиотека не должна запрашивать права, которые не нужны для рендера (файлы только из заданных каталогов, без сетевого доступа по умолчанию).
- **Ввод как недоверенный:** все пути, размеры, ключи кэша и байткод шейдеров, приходящие из конфига, сети или пользователя, считать недоверенными до проверки.
- **Защита в глубину:** даже при сбое одной проверки (например, в драйвере) ограничивать последствия лимитами и изоляцией данных.

### 3.2 Конкретные практики

| Практика | Применение в VAPI |
|----------|-------------------|
| **Валидация входов** | Все пути — canonicalize + jail; размеры буферов/SPIR-V — лимиты; ключи кэша — формат без path traversal. |
| **Жёсткие лимиты** | Максимальный размер буфера, максимальный размер SPIR-V, максимальные размеры текстур в загрузчиках. |
| **Целостность кэша** | Подпись/хеш для файлов кэша шейдеров; отказ в загрузке при несовпадении. |
| **Безопасные значения по умолчанию** | Опции «безопасный режим»: обязательная валидация путей, обязательная проверка SPIR-V, обязательные границы в map/upload. |
| **Аудит зависимостей** | Регулярно проверять GLFW, Vulkan SDK и другие зависимости на известные уязвимости (CVE). |
| **Сборка и дистрибуция** | Не отключать проверки границ в релизе без явного решения; по возможности оставлять опцию валидации SPIR-V для недоверенного контента. |

### 3.3 Чек-лист для интеграции в приложение

- [ ] Задать явный `allowedRoot` (или список корней) для загрузки файлов и шейдеров.
- [ ] Все пути от пользователя/конфига пропускать через `resolveSecure` (или аналог) перед передачей в VAPI.
- [ ] Для пользовательских/сетевых шейдеров включить валидацию SPIR-V и лимит размера.
- [ ] Убедиться, что кэш шейдеров лежит внутри разрешённой директории и при необходимости защищён хешем.
- [ ] В релизе не писать в лог чувствительные данные; путь лог-файла — под контролем и в разрешённом каталоге.
- [ ] Обновлять зависимости и реагировать на CVE.

## 4. Сводная таблица: критичные места и защита

| Критичное место | Риск | Метод защиты |
|-----------------|------|--------------|
| Пути в file_utils, shader, shader_cache, logger | Path traversal, запись в произвольное место | Canonicalize + jail, allowlist расширений, безопасный ключ кэша |
| SPIR-V из файла/кэша | DoS, уязвимости драйвера | Лимит размера, spirv-val, подпись/хеш кэша |
| BufferManager::map(offset, size) | Выход за границы | Проверка offset + size <= buf.size |
| StagingManager::uploadBuffer(dst, data, dstOffset) | Перезапись за границей dst | Проверка dstOffset + data.size() <= dstSize |
| createBuffer(desc) | Исчерпание памяти | Лимит desc.size и/или суммарный лимит |
| Логгер (путь, содержимое) | Утечка, запись в произвольный файл | Jail пути, не логировать чувствительные данные |

## 5. Реализованные меры (код)

В коде добавлено:

- **file_utils:** `resolveSecure(path, allowedRoot)` — канонизация и проверка пути под корнем; `isPathKeySafe(key)` — проверка ключа без `..` и слэшей.
- **BufferManager::map():** проверка `offset < buf.size` и `offset + effectiveSize <= buf.size`; при `VK_WHOLE_SIZE` — `effectiveSize = buf.size - offset`.
- **BufferManager::createBuffer:** отклонение при `desc.size == 0` или `desc.size > 256` MiB.
- **StagingManager::uploadBuffer:** опциональный параметр `dstSize`; при заданном — проверка `dstOffset + data.size() <= *dstSize`.
- **StagingManager::uploadImage / updateImage:** проверка `width/height` в диапазоне 1..16384 и согласованность с `data.size()`.
- **ShaderModule::loadFromFile:** опциональный `allowedRoot`; при заданном — путь разрешается через `resolveSecure`; лимит размера SPIR-V из файла 4 MiB.
- **ShaderModule::loadFromMemory:** лимит размера SPIR-V 4 MiB.
- **FileShaderCache::getOrCompile:** ключ проверяется через `isPathKeySafe`; при непустом `basePath` путь разрешается через `resolveSecure(fullPath, m_basePath)`; лимит размера SPIR-V 4 MiB.
