# План: рефакторинг, ремейк, дебаг и оптимизация

Цель: улучшить поддерживаемость, наблюдаемость и производительность **с сохранением всех текущих возможностей** (обратная совместимость API и поведения, где это не противоречит явным исправлениям).

---

## 1. Дебаг (наблюдаемость и отладка)

### 1.1 Логирование

| Задача | Описание | Сохранение возможностей |
|--------|----------|-------------------------|
| Форматированное логирование | Добавить `logInfo("cat", "value={}", x)` в стиле fmt вместо конкатенации строк (см. [planchanges.md](../planchanges.md) Core). | Существующие `logDebug/Info/Warn/Error/Fatal(cat, msg)` оставить; добавить перегрузки с форматированием опционально (или единый шаблон с variadic args). |
| Уровни и категории | Оставить текущую модель: `LogLevel`, категории, `setMinLevel(category, level)`, `shouldLog()`. | Без изменений контракта. |
| VAPI_LOG_DETAIL | Сохранить опцию: device name, file:line, call-site address. | Без изменений. |
| Синки | Сохранить `ConsoleSink`, `FileSink`, `ILogSink`, `addSink`/`clearSinks`. | Без изменений; при необходимости добавить новый sink (например, ring-buffer для последних N сообщений). |

**Результат:** те же макросы/функции и уровни; при желании — формат-строки без ломания существующих вызовов.

### 1.2 Валидация и проверки (Vulkan / контракты)

| Задача | Описание | Сохранение возможностей |
|--------|----------|-------------------------|
| VK_CHECK / VK_CHECK_LOG | Текущее поведение: `VK_CHECK` возвращает `std::unexpected(toError(_vr))`; `VK_CHECK_LOG` пишет в лог. | Не менять семантику; при рефакторинге вынести общий код в `inline`-функцию с параметром «логировать или только вернуть ошибку». |
| Validation layer | Оставить опцию `enableValidation` в конфиге инстанса; текущий debug messenger. | Без изменений. |
| Assertions | Определить политику: только в Debug, или опция `VAPI_ASSERTS`. Не удалять существующие проверки аргументов. | Сохранить проверки; при необходимости обернуть в макрос, отключаемый в Release. |

### 1.3 Профилирование (CPU / GPU)

| Задача | Описание | Сохранение возможностей |
|--------|----------|-------------------------|
| IProfiler / CpuProfiler | Текущий API: `begin(name)`, `end(name)`, `getElapsedSeconds(name)`, `reset()`. | Без изменений. |
| IGpuProfiler / VkGpuProfiler | Текущий API: `writeTimestamp`, `getResult`, `reset`, `queryCount`. Stub при отсутствии Vulkan. | Без изменений. |
| Интеграция в цикл | Опционально: вызов `profiler.begin("frame")` / `end("frame")` в одном месте (например, run_ui_loop или main loop). | Добавление опционально; не менять существующие точки вызова. |

**Итог по дебагу:** улучшаем формат логов и структуру проверок, не убирая текущие уровни, синки и профилирование.

## 2. Рефакторинг (структура кода без смены поведения)

### 2.1 Логгер и ошибки

- **Logger:** вынести форматирование строки сообщения в отдельную функцию (под будущее fmt); сохранить `LogEntry` и сигнатуры `log(level, category, message, loc, callSiteAddress)`.
- **errorMessage:** уже вынесен в `.cpp`; при добавлении кодов — только правка `error.cpp` и при необходимости тестов.

### 2.2 Модули с высокой связностью

- **FontRenderApp:** уже переведён на `unique_ptr<Impl>`; при дальнейшем рефакторинге — разбивать `Impl` на под-объекты (например, окно/GPU/шрифт) без изменения публичного API.
- **StagingManager:** уже есть persistent staging; при введении полноценного ring buffer — сохранить текущий публичный API и поведение (размеры, перегрузки uploadBuffer/uploadImage).

### 2.3 Именование и типобезопасность

- **Неиспользуемые type aliases** (planchanges): убрать только реально неиспользуемые (`u8fast`, `u16least`, `ptr8`, `cptr16` и т.д.); оставить все, на которые есть ссылки в коде или публичном API.
- **Author hash:** вынести из `types.hpp` в отдельный заголовок или build config; обновить все места включения.

**Принцип:** рефакторинг не меняет публичные контракты и не отключает существующие возможности (логи, валидация, профилирование).

---

## 3. Ремейк (целенаправленная замена компонентов)

Ремейк — замена реализации с сохранением или явной эволюцией контракта.

| Компонент | Текущее состояние | Целевое | Сохранение возможностей |
|-----------|-------------------|--------|--------------------------|
| Emitter (copy on emit) | Уже оптимизирован (копия только ID). | При дальнейшем ремейке (small_vector, lock-free) — сохранить `bind`/`emit`/`disconnect`/`Connection`/`isConnected()`. | Да. |
| Staging | Persistent buffer для одиночных upload; batch по-прежнему создаёт буферы. | Полноценный ring buffer с fence-based recycling (planchanges): тот же API `uploadBuffer`/`uploadImage`/`updateImage` и batch-методы. | Да. |
| Descriptor pool | Уже реализован automatic pool growth. | При ремейке (пул пулов, статистика) — сохранить `createPool`/`allocateSet(s)`/`resetPool`. | Да. |
| BufferManager ID | Уже recycling (free list). | При введении HandlePool — сохранить `BufferId`, `createBuffer`/`destroyBuffer`/`get`/`handle`/`map`. | Да. |

**Правило ремейка:** старый контракт остаётся выполнимым (те же имена, те же гарантии или явно документированные изменения).

## 4. Оптимизация

### 4.1 Уже сделанные (по плану planchanges)

- Connection: weak_ptr; Emitter: копия только ID при emit.
- cmdTransitionImageLayout: fallback TOP/BOTTOM вместо ALL_COMMANDS_BIT.
- Staging: persistent buffer для одиночных upload.
- Descriptor pool growth; BufferManager ID recycling; per-window focus; SizeElement brace-init; ShaderStage унификация; errorMessage в .cpp; FileSink типизированный; FontRenderApp unique_ptr.

### 4.2 Дальнейшие оптимизации (без слома API)

| Область | Действие | Сохранение возможностей |
|---------|----------|-------------------------|
| Layout / виджеты | Size policy, size hints (planchanges UI) — добавить методы, не убирать текущие `setGeometry`/`geometry()`. | Да. |
| Draw batching | Батчинг draw calls на уровне рендера — внутренняя реализация; публичный API отрисовки сохранить. | Да. |
| VMA | Интеграция Vulkan Memory Allocator для буферов/изображений — замена внутренней аллокации; `BufferManager`/`ImageManager` контракты сохранить. | Да. |
| Pipeline cache | Уже есть; инвалидация по UUID/версии драйвера (planchanges) — расширение, не замена. | Да. |

### 4.3 Benchmark suite (planchanges, инфраструктура)

- Добавить отдельный target (например, `bench_staging`, `bench_layout`, `bench_draw`) с тем же набором заголовков и библиотек.
- Метрики: throughput staging, время layout, количество draw calls за кадр.
- Не менять production-код ради бенчмарков; по возможности выносить сценарии в отдельные файлы.

**Итог по оптимизации:** все изменения — либо внутренние, либо расширение API (новые методы/опции), без удаления существующих возможностей.

## 5. Оценка необходимости реинжиниринга

Реинжиниринг здесь понимается как **существенное перепроектирование подсистемы или архитектуры** (смена контрактов, разрыв обратной совместимости, замена стека).

### 5.1 Когда реинжиниринг **не** нужен

- Локальные баги, точечные оптимизации, добавление форматирования в лог, новые опции (например, ещё один sink или уровень логирования).
- Рефакторинг внутри реализации при сохранении публичного API.
- Добавление новых модулей (например, анимации, tooltips) без переписывания существующих.

### 5.2 Когда реинжиниринг **стоит рассматривать**

| Критерий | Вопрос | Порог |
|----------|--------|--------|
| Техдолг | Насколько часто изменения в одном месте ломают неочевидные места? | Высокий темп поломок при правках — повод выделить модуль и переписать его контракт и реализацию. |
| Производительность | Есть ли измеримые требования (FPS, latency, throughput), которые не достигаются точечными оптимизациями? | Если да — реинжиниринг узкого места (например, рендер-путь или ресурсный менеджер). |
| Масштаб | Планируется ли второй бэкенд (D3D/Metal), вторая платформа (SDL, headless), многопоточный рендер? | Если да — реинжиниринг абстракций (GPU, платформа, команды) может окупиться. |
| Поддерживаемость | Понимают ли новые разработчики код за приемлемое время? | Если нет и рефакторинг «по кускам» не помогает — реинжиниринг модуля с документированием. |

### 5.3 Процедура принятия решения

1. **Зафиксировать цель:** что именно не устраивает (производительность, поддержка, расширяемость).
2. **Оценить объём:** точечный рефакторинг vs замена целого модуля vs смена архитектуры.
3. **Проверить достаточность малых шагов:** можно ли достичь цели рефакторингом + ремейком без полного реинжиниринга.
4. **Если реинжиниринг выбран:** определить границы (один модуль, слой, подсистема), сохранить совместимость на границах (фасад, адаптер) или явно версионировать API.

### 5.4 Текущая рекомендация по проекту

- **Сейчас:** рефакторинг и точечный ремейк (лог, профилирование, оптимизации из planchanges) **достаточны**; полный реинжиниринг не обязателен.
- **Рассматривать реинжиниринг** при: появлении второго GPU-бэкенда или второй платформы, либо при жёстких требованиях к производительности, которые не закрываются VMA, ring buffer, батчингом и профилированием.

## 6. Чек-лист сохранения возможностей

Перед любым изменением в рамках плана проверять:

- [ ] Все существующие публичные API (классы, методы, константы), используемые в примерах и документации, сохранены или явно помечены deprecated с заменой.
- [ ] Уровни логирования и категории, синки (Console, File), макросы VAPI_LOG_DETAIL работают как раньше.
- [ ] Vulkan validation и VK_CHECK/VK_CHECK_LOG ведут себя как раньше (ошибки возвращаются или логируются).
- [ ] CPU/GPU профилирование (IProfiler, IGpuProfiler) и их использования в main/тестах не сломаны.
- [ ] Тесты (unit, integration) проходят после изменений; при необходимости добавлены тесты на новое поведение.
- [ ] Документация (quick_reference, detail_docs, testing.md) обновлена при изменении API или поведения.

---

## 7. Порядок работ (предложенный)

1. **Дебаг:** форматированное логирование (опциональные перегрузки), затем при необходимости ring-buffer sink для последних N сообщений.
2. **Рефакторинг:** вынести author hash; при необходимости — неиспользуемые type aliases; внутренняя структура Logger (форматирование).
3. **Оптимизация:** по приоритету из planchanges (VMA, ring staging, батчинг) с сохранением контрактов.
4. **Benchmark suite:** отдельный target и сценарии без изменения production-пути.
5. **Оценка реинжиниринга:** периодически (например, раз в мажорный цикл) проходить по критериям из раздела 5 и фиксировать решение в CHANGELOG или отдельном ADR.

Документ можно использовать как основу для тикетов и ревью: каждое изменение привязывать к пункту плана и к чек-листу раздела 6.
